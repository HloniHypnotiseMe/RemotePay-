# RemotePay System Architecture\n\n## Overview\n\nRemotePay is a South African payment platform built on PayGate API. The system is architected using a layered approach with separation of concerns.\n\n## Architecture Diagram\n\n```\n┌─────────────────────────────────────────────────────────────┐\n│                    Client Applications                       │\n│         (Merchant Dashboard, Web, Mobile)                    │\n└────────────────────┬────────────────────────────────────────┘\n                     │\n        ┌────────────┴────────────┐\n        │                         │\n   ┌────▼──────────┐     ┌──────▼────────┐\n   │  HTTP/HTTPS   │     │  Rate Limit   │\n   └────┬──────────┘     │  CORS         │\n        │                │  Helmet       │\n        │                └──────┬────────┘\n        │                       │\n   ┌────▼───────────────────────▼────────────┐\n   │         Express.js Server (3000)        │\n   │                                          │\n   │  ┌─────────────────────────────────┐   │\n   │  │  Routing Layer                  │   │\n   │  │  - Auth Routes                  │   │\n   │  │  - Merchant Routes              │   │\n   │  │  - Transaction Routes           │   │\n   │  │  - Subscription Routes          │   │\n   │  │  - Webhook Routes               │   │\n   │  └──────────┬──────────────────────┘   │\n   │             │                           │\n   │  ┌──────────▼──────────────────────┐   │\n   │  │  Middleware Layer               │   │\n   │  │  - Authentication               │   │\n   │  │  - Authorization (RBAC)         │   │\n   │  │  - Validation                   │   │\n   │  │  - Error Handling               │   │\n   │  └──────────┬──────────────────────┘   │\n   │             │                           │\n   │  ┌──────────▼──────────────────────┐   │\n   │  │  Controllers & Services         │   │\n   │  │  - Auth Service                 │   │\n   │  │  - Merchant Service             │   │\n   │  │  - Transaction Service          │   │\n   │  │  - PayGate Service              │   │\n   │  │  - Webhook Service              │   │\n   │  └──────────┬──────────────────────┘   │\n   │             │                           │\n   │  ┌──────────▼──────────────────────┐   │\n   │  │  Utilities & Helpers            │   │\n   │  │  - Crypto (Signatures)          │   │\n   │  │  - Logger                       │   │\n   │  │  - Validators                   │   │\n   │  │  - QR Generator                 │   │\n   │  └──────────┬──────────────────────┘   │\n   └─────────────┼────────────────────────────┘\n                 │\n        ┌────────┴────────┐\n        │                 │\n   ┌────▼────────┐   ┌───▼──────────┐\n   │ PostgreSQL  │   │  PayGate API  │\n   │   (5432)    │   │ (secure.     │\n   │             │   │  paygate.    │\n   │ - Users     │   │  co.za)      │\n   │ - Merchants │   │              │\n   │ - Trans.    │   │ - PayWeb     │\n   │ - Subs.     │   │ - PaySubs    │\n   │ - Logs      │   │ - Webhooks   │\n   └─────────────┘   └──────────────┘\n```\n\n## Layers\n\n### 1. Presentation Layer\n\n- HTTP endpoints\n- Rate limiting\n- CORS handling\n- Request/Response formatting\n\n### 2. Routing Layer\n\n- API routes (`/auth`, `/merchants`, `/transactions`, etc.)\n- Route protection (authentication, authorization)\n- Request validation\n\n### 3. Middleware Layer\n\n- **Authentication**: JWT validation\n- **Authorization**: RBAC (Admin, Merchant, Support, Customer)\n- **Validation**: Request body validation with Joi\n- **Error Handling**: Global error handler\n- **Logging**: Request/Response logging\n\n### 4. Business Logic Layer\n\n**Controllers**: Handle HTTP requests, delegate to services\n\n**Services**: Core business logic\n- AuthService\n- MerchantService\n- TransactionService\n- SubscriptionService\n- PayGateService\n- WebhookService\n- PaymentLinkService\n\n### 5. Data Layer\n\n**Database**: PostgreSQL\n- Connection pooling\n- Query execution\n- Transaction management\n\n**Models**: Entity definitions\n- User\n- Merchant\n- Transaction\n- Subscription\n- PaymentLink\n- AuditLog\n- WebhookEvent\n- Settlement\n\n### 6. Utilities Layer\n\n- **Logger**: Pino logging\n- **Crypto**: JWT, signatures, encryption\n- **Validators**: Input validation\n- **QR Generator**: QR code generation\n- **Helpers**: Common utilities\n\n## Data Flow\n\n### Payment Flow\n\n```\n1. Merchant initiates payment\n   POST /transactions/initiate\n   ↓\n2. Transaction created (pending)\n   Database insert\n   ↓\n3. PayGate request prepared\n   Reference, amount, merchant ID\n   ↓\n4. Redirect URL generated\n   Return to client\n   ↓\n5. Customer enters payment details on PayGate\n   ↓\n6. PayGate processes payment\n   ↓\n7. PayGate sends webhook\n   POST /webhooks/paygate\n   ↓\n8. Signature verified\n   ↓\n9. Transaction status updated\n   Database update\n   ↓\n10. Merchant notified (email/webhook)\n   ↓\n11. Settlement prepared\n    Reconciliation\n```\n\n### Subscription Flow\n\n```\n1. Merchant creates subscription\n   POST /subscriptions\n   ↓\n2. Subscription record created\n   PaySubs registration\n   ↓\n3. Initial charge attempted\n   ↓\n4. Recurring charges on schedule\n   PayGate PaySubs API\n   ↓\n5. Webhooks received for each charge\n   ↓\n6. Transaction records created\n   ↓\n7. Failures handled with retry logic\n   ↓\n8. Subscription status updated\n```\n\n## Security Architecture\n\n### Authentication\n\n- **JWT Bearer tokens**: Issued on login\n- **Refresh tokens**: Long-lived, used to get new access tokens\n- **Token expiration**: 24 hours for access, 7 days for refresh\n- **Secure storage**: Tokens in httpOnly cookies or secure storage\n\n### Authorization\n\n- **Role-Based Access Control (RBAC)**: Admin, Merchant, Support, Customer\n- **Merchant isolation**: Merchants can only access their own data\n- **Audit logging**: All actions logged\n\n### Data Security\n\n- **No card storage**: PayGate handles card details\n- **Encryption**: Sensitive data encrypted at rest\n- **HTTPS**: All communication encrypted in transit\n- **PCI DSS compliance**: No storage of card data\n\n### API Security\n\n- **Rate limiting**: 100 requests per 15 minutes\n- **Input validation**: All inputs validated\n- **Helmet.js**: Security headers\n- **CORS**: Restricted origins\n- **Signature verification**: PayGate webhooks verified\n\n## Deployment Architecture\n\n### Docker Setup\n\n```\n┌─────────────────────────────────┐\n│      Docker Compose Network     │\n│                                 │\n│ ┌────────────┐  ┌────────────┐ │\n│ │   Nginx    │  │  API Node  │ │\n│ │ (Proxy)    │→ │ (3000)     │ │\n│ └────────────┘  └──────┬─────┘ │\n│                        │       │\n│                ┌───────▼────┐  │\n│                │ PostgreSQL │  │\n│                │  (5432)    │  │\n│                └────────────┘  │\n└─────────────────────────────────┘\n        ↓\n  External Services\n  - PayGate API\n  - External Webhooks\n```\n\n### Scaling Considerations\n\n- **Horizontal**: Multiple API instances behind load balancer\n- **Database**: Connection pooling (min: 5, max: 20)\n- **Caching**: Redis for session/cache (future)\n- **Queue**: Async job processing (future)\n\n## Error Handling\n\nAll errors follow standard format:\n\n```json\n{\n  \"success\": false,\n  \"error\": {\n    \"code\": \"ERROR_CODE\",\n    \"message\": \"User-friendly message\",\n    \"details\": {}\n  },\n  \"timestamp\": \"2024-01-15T10:30:00Z\"\n}\n```\n\n## Logging Strategy\n\n- **Level**: debug, info, warn, error\n- **Format**: JSON in production, pretty in development\n- **Fields**: timestamp, level, message, context\n- **Sensitive data**: Masked in logs\n\n## Performance Optimization\n\n- **Database indexing**: On frequently queried columns\n- **Connection pooling**: Reuse database connections\n- **Response compression**: Gzip enabled\n- **Rate limiting**: Prevent abuse\n- **Caching headers**: HTTP caching for static responses\n\n## Monitoring & Observability\n\n- **Health check**: `/health` endpoint\n- **Uptime monitoring**: External monitoring service\n- **Error tracking**: Centralized error logging\n- **Performance metrics**: Response time, request count\n- **Audit logs**: User actions and changes\n\n## Future Architecture Enhancements\n\n1. **Message Queue**: Redis/RabbitMQ for async processing\n2. **Caching Layer**: Redis for session and data caching\n3. **Microservices**: Separate services for different domains\n4. **Event Streaming**: Kafka for event-driven architecture\n5. **API Gateway**: Kong or similar for advanced routing\n6. **Service Mesh**: Istio for service-to-service communication\n7. **Multi-region**: Deployment across multiple regions\n8. **Database Replication**: Master-slave for high availability\n\n## Technology Stack\n\n| Component | Technology |\n|-----------|------------|\n| Language | TypeScript |\n| Runtime | Node.js 18+ |\n| Framework | Express.js |\n| Database | PostgreSQL 15 |\n| Authentication | JWT |\n| Validation | Joi |\n| Logging | Pino |\n| Testing | Jest |\n| Containerization | Docker |\n| Orchestration | Docker Compose |\n| CI/CD | GitHub Actions |\n| Code Quality | ESLint, Prettier |\n"
